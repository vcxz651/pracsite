## 0. 문서 목적

이 문서는 현재 데모 페이지의 단일 기획 기준이다.
예전 `시나리오 1/2/3 + /tutorial/ 연계` 안은 폐기했고, 지금은 **인트로 페이지 + 시나리오 A 단일 체험 + `/demo/tutorial/` 인터랙티브 기능 튜토리얼** 구조를 기준으로 삼는다.
2026-03-01 기준 공개 체험 진입은 시나리오 A 단일 흐름으로 고정하고, 시나리오 B/C는 레거시 내부 경로로만 남긴다.

---

## 1. 핵심 목표

1. 빈 화면 없이 의미 있는 더미 상태를 바로 보여준다.
2. 데모 전용 축소판이 아니라, 실제 핵심 기능을 거의 그대로 체험하게 한다.
3. 기본 체험 흐름은 매니저 관점 핵심 화면에 집중한다. 역할 전환은 주 체험 경로로 쓰지 않는다.
4. 기능 설명은 투어로 하되, 기능 사용 자체는 막지 않는다.
5. 데모 진입 직후에는 인트로 페이지에서 먼저 복잡도를 체감시키고, 단일 CTA로 바로 시나리오 A 체험에 들어가야 한다.

---

## 2. 현재 방향 (2026-03-01)

- `/demo/` 진입 시 인트로 페이지를 먼저 보여준다.
- 인트로 페이지 첫 화면에서 **A 시나리오의 복잡도(40명, 50곡, 곡당 주 60분 합주)** 를 즉시 보여준다.
- 그 아래에서 단일 체험 버튼(`딸깍`)으로 바로 시나리오 A를 시작한다.
- 시나리오 B/C는 공개 체험 기본 동선에서 제거하고, 레거시 내부 경로로만 유지한다.
- 시나리오 A 본편은 자동 매칭 보드 체험이고, 기능 설명은 별도 `/demo/tutorial/` 인터랙티브 튜토리얼에서 처리한다.
- `/demo/tutorial/`은 정적 슬라이드가 아니라, 실제 `meeting_detail` 패턴을 축소 재현한 5곡 고정 데이터셋 체험형 페이지다.
- 튜토리얼은 spotlight tour로 현재 단계의 핵심 조작만 강조하고, 나머지 UI는 그대로 만질 수 있게 둔다.
- 추가합주 기능은 홈 인트로 영상으로 이미 설명하므로 이번 데모 핵심 시나리오에서 제외한다.
- 연합 공연 기능은 아직 구현 범위가 커서 데모 시나리오에 넣지 않는다.

## 3. 시나리오 구성

### 시나리오 A: 기본 자동 매칭 체험
- 자동 매칭이 완료되어 보드에 카드가 충분히 배치된 상태
- 드래그앤드롭, 리모콘 필터, 임시 저장, 공유 직전까지 체험
- 목적: “합주 시간표가 빠르게 나온다”는 핵심 가치 전달
- 기본 데모 흐름은 **배정까지 이미 끝난 상태**에서 시작하고, 사용자는 `매칭하기`부터 바로 체험하게 한다.
- 시나리오 A는 체감 기능을 강조하기 위해 **매니저 관점 자동 매칭 보드로만 진입**한다. 역할 전환 UI가 남아 있어도 시나리오 A 진입 결과는 `meeting_detail`이 아니라 자동 매칭 화면 기준으로 유지한다.
- `모든 곡 임의 배정` 같은 테스트용 버튼은 주 흐름으로 쓰지 않는다. 남겨두더라도 보조/테스트 경로로만 취급한다.

### 시나리오 B (레거시 내부 경로)
- 공유 전 또는 공유 직전 조율 상태
- 강제배치/미배치/임시합주실이 일부 포함된 상태
- 목적: “어려운 케이스도 조율 가능하다”를 보여줌

### 시나리오 C (레거시 내부 경로): 공유 후 변경 정리 + 예약 확정
- 이미 한 번 공유된 임시 시간표에서 일부 수정이 필요한 상태
- `공유 후 변경만`, `강제 배치만`, `임시합주실만` 같은 매니저 도구를 활용
- 예약 단계 진입 후 예약 필요 목록 확인, 예약 임시 저장, 예약 확정까지 이어짐
- 목적: 운영자 관점의 변경 관리와 마무리 흐름 전달

### 폐기된 예전 방향
- `시나리오 3 = 최종 확정 완료 + 추가합주 체험`
- `/tutorial/`로 넘기는 전체 튜토리얼 CTA
- 기술 상태(DRAFT/BOOKING/FINAL)만 앞세운 설명 방식

## 4. 더미 데이터 기준

### 저장 전략
- 데모 데이터는 사용자 클릭 시 매번 통째로 생성하지 않는다.
- 서버 DB에 **시나리오별 데모 템플릿**을 미리 저장해 둔다.
- 기본 원칙은 템플릿 직접 공유가 아니라 **작업용 미팅(working copy)** 복제 진입이다.
- 예외: 시나리오 A는 2026-02-28 기준으로 **단일 공유 템플릿을 그대로 사용**한다. 로딩 지연 없이 즉시 자동 매칭을 보여주기 위한 운영 예외다.
- 세션에는 실제 데이터 본문이 아니라 `demo_mode`, `demo_role`, `demo_scenario`, `demo_meeting_id` 같은 **참조값만** 저장한다.

운영 규칙:
- 템플릿은 lazy create도 가능하지만, 실제 시연 전에는 관리 커맨드로 **선생성(prewarm)** 하는 것을 기본으로 한다.
- 권장 커맨드: `./.venv/bin/python manage.py prepare_demo_cache`
- 운영/배포 환경에서는 권장이 아니라 **배포 직후 선실행 권장도가 매우 높다**. 시나리오 A/C 첫 진입은 템플릿이 없으면 요청 한 번에 더미 생성 + 가용성 동기화가 몰려 `WORKER TIMEOUT`으로 실패할 수 있다.
- 실제 운영 서버에서는 배포 직후 또는 DB 초기화 직후 `python manage.py prepare_demo_cache`를 먼저 실행한 뒤 `/demo/start/`를 열어야 한다.
- 데모 템플릿/더미 생성 로직을 수정했으면, `manage.py check`만으로 끝내지 말고 `prepare_demo_cache` 또는 실제 `/demo/start/` 진입까지 한 번 확인한다.
- 서버 재시작이나 일반 코드 배포만으로는 템플릿이 사라지지 않는다.
- DB 초기화/리셋 후에는 `prepare_demo_cache`를 다시 실행해야 한다.

### 공통
- 밴드명: `[데모] 락스타즈`
- 곡 템플릿 CSV: `김민기_밴드음악_명곡선_선곡템플릿.csv` 단일 기준
- 멤버 수와 곡 수는 시나리오별로 다르다.
- 합주실: `A룸 (3F)`, `B룸 (3F)`
- 필요 시 시나리오 B/C에서 임시합주실 추가

### 시나리오 A
- `MeetingFinalDraft` 없음 (공유 전)
- `MeetingWorkDraft`도 남기지 않는다. `load_saved` 기반 프리로드 초안은 금지한다.
- 멤버 수: 40명
- 곡 수: 50곡
- 50곡 모두 일반 세션이 있어야 한다.
- 50곡 모든 일반 세션에 assignee가 있어야 한다.
- 시나리오 A 템플릿 곡은 `needed + extra` 전체 세션 수가 6 이하인 곡만 사용한다.
- `extra_session`도 빈칸으로 두지 않고, 시나리오 A에서는 assignee와 applicant가 기본으로 채워진 상태여야 한다.
- 곡당 기본 합주 시간은 60분(2슬롯) 기준이다.
- 초기 인트로 화면은 실제 시나리오 A 템플릿 데이터를 그대로 프리뷰한다.
- `딸깍` 후에는 저장된 초안을 로드하지 않고, `force_rematch=1`로 **실제 자동 매칭을 즉시 다시 실행**해야 한다.
- 초기 배정 더미는 고정 시드 기반으로 재현되며, 드럼 배정은 편차가 과하게 벌어지지 않도록 균등 분배 우선으로 생성한다.

### 시나리오 B
- `is_booking_in_progress=False`
- 멤버 수: 6명
- 곡 수: 20곡
- 강제배치 카드 1건 이상
- 임시합주실 카드 1건 이상
- 미배치 카드 1~2건 포함 가능

### 시나리오 C
- `MeetingFinalDraft.events` 존재
- `is_final_schedule_released=True`
- 멤버 수: 40명
- 곡 수: 80곡
- 시작 시점은 `is_booking_in_progress=False`
- 공유 이후 변경된 카드가 몇 건 존재
- 예약 단계 진입 후에는 `booking_completed_keys` 일부만 완료된 상태로 이어지게 구성

## 5. 데모 진입 UX

### 시작 흐름
- `/demo/` 진입 시 가장 먼저 **인트로 페이지**를 보여준다.
- 인트로 페이지에는 복잡도 체감 요소(고정 40명 이름 전광판, 확정된 50곡 목록, `딸깍` CTA)를 배치한다.
- 사용법 안내는 인트로에서 길게 설명하지 않고, 필요하면 별도 튜토리얼 페이지(`/demo/tutorial/`)로 분리한다.
- 사용자는 먼저 “이걸 손으로 맞추기 어렵다”를 체감한 뒤, 기본적으로 시나리오 A에 바로 진입한다.
- 현재 인트로는 시나리오 선택 탭 없이 단일 CTA만 제공한다.

### 진입 후 동작
- 인트로 페이지 → `딸깍` → DB 템플릿 확인/준비 → 시나리오 A 자동 매칭 화면 진입
- 첫 진입 시에는 화면을 바로 만져볼 수 있도록 투어를 자동 시작하지 않는다.
- 필요할 때만 상단 배너의 `기능 튜토리얼` 버튼으로 `/demo/tutorial/`을 연다.
- 첫 인트로 페이지(`/demo/`)에서는 스크롤 하단 모달 같은 부가 안내를 띄우지 않는다.

## 6. 투어 원칙

- 투어는 **안내 전용**이다.
- 투어가 기능 사용을 막아서는 안 된다.
- 주요 UI 요소를 스팟라이트하고 짧은 설명만 제공한다.
- 현재 기본 체험은 매니저 관점 기준으로 설명한다. 멤버 뷰 분기는 남아 있어도 레거시/디버그 보조 경로로 본다.
- `/demo/tutorial/`의 spotlight는 `STEP 1 환영`, `STEP 2 세션 지원`, `STEP 3 세션 지원 마감`까지만 짧게 안내하고, 이후 단계 진행은 사용자가 직접 조작하게 둔다.

### 기본 설명 대상 예시
- 스케줄 보드
- 곡 카드
- 리모콘 필터
- 공유 버튼
- 예약 필요 목록
- 역할 전환 배너

### `/demo/tutorial/` 현재 구현 기준
- 튜토리얼은 `ㄷㅅㅎㅅ`, `Can't We Start It All Over Again`, `비구름`, `A`, `마냥 걷는다` 5곡을 고정 순서로 사용한다.
- STEP 1은 **세션 지원 단계**다. 배정자 이름은 보여주지 않고, `ㄷㅅㅎㅅ`의 `G1`만 직접 지원/취소할 수 있다.
- STEP 1에서 세션 버튼을 누르면 지원자 수 배지, 지원자 목록, 카드 배경색이 함께 바뀐다.
- STEP 2와 STEP 3에서만 배정자 이름이 보인다.
- STEP 1, STEP 2 모두 각 곡 카드마다 `지원자 펼치기/접기` 버튼이 있고, 해당 곡의 세션별 지원자 목록을 카드 내부에서 토글한다.
- 튜토리얼용 지원자 표시는 실제 미팅 데이터를 우선 사용하되, 표시 밀도를 위해 참가자 이름으로 보강할 수 있다. 이 표시는 데모 가독성용이며 실제 저장 데이터 변경이 아니다.
- 튜토리얼 내부 조작은 실제 저장/API 호출 없이 프론트 상태만 바꾼다.

## 7. 세션 데이터 구조

```python
request.session['demo_mode'] = True
request.session['demo_role'] = 'manager'          # 레거시 전환 경로에서만 'member'
request.session['demo_scenario'] = 1              # 1, 2, 3
request.session['demo_band_id'] = '<uuid>'
request.session['demo_meeting_id'] = '<uuid>'
request.session['demo_user_manager_id'] = '<pk>'
request.session['demo_user_member_id'] = '<pk>'
```

규칙:
- 시나리오 템플릿은 서버 DB에 상시 보관할 수 있다.
- 기본적으로 실제 데모 체험은 템플릿 원본이 아니라 **작업용 미팅 복제본**으로 진행한다.
- 예외적으로 시나리오 A는 단일 공유 템플릿을 그대로 사용한다.
- 세션은 원본 데이터를 담지 않고, 현재 작업용 미팅과 역할을 가리키는 포인터만 저장한다.
- 역할 전환은 `demo_role`과 데모 전용 로그인 유저 스왑으로 처리하지만, 현재는 레거시/디버그 보조 경로로만 유지한다.
- 공개 체험 시작 경로(`demo_home -> demo_start`)는 시나리오 A 단일 흐름으로 본다.

## 8. 현재 구현 체크포인트

1. 공개 진입은 `/demo/` 인트로 + 시나리오 A 단일 CTA다.
2. 시나리오 A 본편은 자동 매칭 보드 체험이다.
3. 기능 설명은 `/demo/tutorial/` 인터랙티브 튜토리얼로 분리한다.
4. 시나리오 B/C는 레거시 내부 경로로만 남긴다.
5. 역할 전환은 핵심 UX가 아니라 레거시/디버그 보조 경로로 본다.
6. 데모용 조작은 실제 저장 대신 프론트 연출 또는 읽기 전용으로 유지한다.

### 임시 노출 기능 정리 우선순위

현재는 테스트 편의를 위해 아래 버튼을 남겨둘 수 있지만, 장기적으로는 일반 화면/데모에서 숨기거나 제거하는 것을 기본 방향으로 본다.

1. 높음: `모든 곡 임의 지원`
2. 높음: `모든 지원 초기화`
3. 높음: `모든 곡 임의 배정`

정리 원칙:
- 테스트가 끝나면 일반 사용자 경로에서 숨기고, 필요하면 관리자 전용 개발 보조 기능으로만 남긴다.
- 데모에서는 시나리오 핵심 흐름과 직접 관련이 없으면 노출하지 않는 쪽을 우선한다.
- 완전 삭제 전까지는 “임시 테스트용”이라는 성격을 문서에서 유지한다.

## 9. 운영 주의사항

- `match_result.html`과 `meeting_detail.html`은 실제 화면을 재사용한다.
- 데모 전용 권한 바이패스 로직을 새로 만들지 않는다.
- `is_demo`는 배너/모달/투어 노출 제어에만 사용한다.
- 실제 권한은 `request.user`와 기존 permission 함수 기준으로 유지한다.
- 데모에서는 시나리오 핵심 흐름에 없는 관리 기능(`참가자 관리`, `합주 정보 수정`)은 숨기고, 서버에서도 읽기 전용으로 막는다.
- `모든 곡 임의 지원`, `모든 지원 초기화`, `모든 곡 임의 배정`은 현재 테스트용 임시 노출 기능으로 본다. 추후 테스트 단계가 끝나면 숨김 또는 제거를 우선 검토한다.
- 템플릿 원본 미팅은 사용자 행동으로 오염되면 안 된다. 저장/수정/예약은 작업용 미팅에서만 일어나야 한다.
- 예외적으로 시나리오 A는 템플릿 원본을 직접 보지만, **저장 경로는 전부 닫혀 있어야 한다.**
- 시나리오 A에서는 `MeetingWorkDraft` 저장, 최종 저장, 예약 저장, `load_saved` 재진입을 허용하지 않는다.
- 시나리오 A는 항상 저장 초안 재생이 아니라 **실제 자동 매칭 재실행 결과**를 보여줘야 한다. 미리 채운 `MeetingWorkDraft.events`를 불러와 강제배치 오해를 만들면 안 된다.
- 시나리오 A는 멤버로 전환된 세션에서도 진입 결과를 자동 매칭 화면으로 유지한다. `meeting_detail` 우회 진입으로 체험 흐름을 바꾸지 않는다.
- 데모 배너의 역할 전환 UI는 현재 기본 노출 대상이 아니다. 핵심 체험 경로는 비로그인 진입 -> 매니저 관점 화면 진입으로 본다.
- 매칭 보드에서 `기능 튜토리얼`을 누르면, 실제 `meeting_detail` 원본으로 가지 않고 인터랙티브 전용 `demo/tutorial/` 페이지로 이동한다.
- 튜토리얼은 `STEP 1 세션 지원`, `STEP 2 배정/강제 배정`, `STEP 3 배정 리모콘` 3단계로 구성하고, 마지막에 실제 `match/settings`로 이동시킨다.
- `STEP 1`은 배정 전 단계처럼 보여야 하므로, 배정자 이름을 노출하지 않는다.
- `STEP 1`은 별도 설명 버튼 대신 곡 카드 안 세션 버튼 클릭(지원 상태 변화)과 `세션 지원 마감` 버튼으로 직접 다음 단계로 넘기는 연출을 사용한다.
- 튜토리얼 셋리스트는 데모 미팅의 실제 곡 `ㄷㅅㅎㅅ`, `Can't We Start It All Over Again`, `비구름`, `A`, `마냥 걷는다` 5장을 고정 순서로 사용한다.
- 현재는 “정적 목업”이 아니라, 실제 `meeting_detail` 패턴을 축소한 인터랙티브 셋리스트로 보는 것이 맞다.
- 튜토리얼 내부 지원자 칩 레이아웃(중앙 정렬, 이름만 표시)은 `docs/demo_tutorial_applicant_chip_layout.md`에 보관하고, 추후 `meeting_detail` 실제 UI 개선 때 참고한다.
- 초기 장식형 CSS는 런타임에서 제거하고, 보관 메모만 `docs/demo_feature_tutorial_css_archive.md`에 남긴다.
- 시나리오 C는 “공유 후 변경 정리 + 예약 확정”에 집중한다. 연합 공연/추가합주 데모로 확장하지 않는다.
- 운영 서버에서 데모를 시연할 일이 있으면, 배포 직후 `prepare_demo_cache`를 먼저 실행해 공개 시나리오 A와 레거시 B/C 템플릿을 선생성한다.

### 2026-02-28 회고 메모
- 시나리오 A에서 미리 만든 `MeetingWorkDraft.events`를 `load_saved=1`로 띄우면, 실제 자동 매칭 결과가 아니라 단순 슬롯 배치 초안이 렌더링될 수 있다.
- 이 상태에서 매칭 뷰가 `_recompute_forced_flags(...)`를 다시 돌리면, 실제 가용성과 안 맞는 카드가 `강제배치`로 바뀌어 체험 신뢰도를 해친다.
- 따라서 시나리오 A의 핵심 원칙은:
  - 인트로 프리뷰는 실제 템플릿 데이터 기반
  - 진입 후 보드는 저장 초안 재생 금지
  - `force_rematch=1` 기반 실제 자동 매칭 실행
  - 저장 API 전부 차단

### 알려진 경미 이슈
- 데모에서 첫 페이지 전환이 아직 처리 중일 때 다른 전환/요청을 겹치면, 일부 브라우저/웹뷰 환경에서 `Load failed` 메시지가 남을 수 있다.
- 현재는 중복 요청 방어를 넣어 빈도를 줄였지만, 완전히 제거되지는 않았다.
- 기능 진행 자체는 가능한 경우가 많아 우선순위는 낮게 본다. 데모 핵심 기능 버그로 분류하지 않는다.

### 알려진 운영 이슈
- 운영 서버에서 `prepare_demo_cache` 없이 시나리오 A/C를 처음 열면, lazy create 경로가 한 요청 안에서 대량 더미 데이터를 생성하면서 Gunicorn `WORKER TIMEOUT`으로 `/demo/start/`가 `500 Internal Server Error`로 실패할 수 있다.
- 이 이슈는 템플릿 데이터가 한 번 생성된 뒤에는 재현 빈도가 크게 낮아진다. 따라서 운영 배포 체크리스트에서는 `prepare_demo_cache` 선실행을 사실상 필수 취급한다.
