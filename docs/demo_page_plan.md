## 0. 튜토리얼과 데모의 차이

이 문서가 다루는 것은 **데모 페이지**다. 기존 `tutorial_plan.md`의 튜토리얼과 목적이 다르다.

| 항목 | 튜토리얼 (`tutorial_plan.md`) | 데모 페이지 (이 문서) |
|---|---|---|
| 대상 | 신규 유저, 학습 목적 | 평가자·발표 수신자·관심 있는 사람 |
| 목적 | 단계별 학습, 기능 체득 | 서비스 핵심 가치를 빠르게 인식 |
| 흐름 | 처음부터 끝까지 직접 진행 | 미리 세팅된 풍부한 상태에서 탐색 |
| 데이터 | 빈 상태에서 직접 채워 나감 | 처음부터 의미 있는 더미 데이터 제공 |
| 역할 | 단일 역할(주로 매니저) | **매니저 / 일반 멤버 전환 가능** |
| 기간 | 6단계 순서 진행 | 시나리오별 자유 탐색 |

두 기능은 서로 보완 관계다. 데모 페이지를 먼저 보고 흥미를 느낀 사람이 튜토리얼로 이동하는 동선을 설계한다.

---

## 1. 데모의 핵심 목표

1. **더미 데이터를 미리 제공한다** — 빈 화면을 절대 노출하지 않는다.
2. **역할 배너를 항상 표시한다** — "지금 매니저 관점의 화면입니다" / "지금 일반 멤버 관점의 화면입니다".
3. **사용자가 직접 모든 기능을 시연할 수 있다** — 튜토리얼 이후 자유 탐색 모드.
4. **튜토리얼은 기능을 설명하고 안내하되 강제하지 않는다** — 건너뛰기 가능.

---

## 2. 데모 시나리오 전체 구조

데모는 **3개의 사전 세팅 시나리오**를 제공한다. 각 시나리오는 Meeting의 다른 상태를 보여준다.

```
/demo/
├── /demo/start/          # (POST) 데모 데이터 초기화 → 시나리오 선택 화면
├── /demo/scenario/<N>/   # 시나리오 N 진입 (1~3)
├── /demo/switch-role/    # (POST) 역할 전환 (매니저 ↔ 일반 멤버)
└── /demo/exit/           # (POST) 데이터 삭제 + 세션 정리
```

### 시나리오 1: 조율 중 (DRAFT_MATCHING)
- 자동 매칭이 완료되어 보드에 카드가 배치된 상태
- 사용자가 드래그앤드롭으로 조율 → 공유까지 체험
- **매니저 뷰**: 보드 편집 가능, 리모콘 필터 사용 가능, 공유 버튼 활성
- **멤버 뷰**: 읽기 전용, 아직 공유 전 상태 안내

### 시나리오 2: 예약 진행 중 (BOOKING_IN_PROGRESS)
- 공유가 완료된 상태. 합주실 예약 확정 단계
- 강제배치 1건 + 임시합주실 1개가 이미 포함된 데이터
- **매니저 뷰**: 예약 체크 완료, 강제배치 경고 모달, 임시합주실 이름 확정 가능
- **멤버 뷰**: 공유된 일정표 확인, 불가능 오버레이 및 사유 보기

### 시나리오 3: 최종 확정 완료 (FINAL_CONFIRMED)
- 모든 과정이 끝난 상태. 추가 합주 기능 체험 가능
- **매니저 뷰**: 최종 일정표 + 추가합주(+합주) 버튼 전체 활성
- **멤버 뷰**: 확인(ack) 완료 상태, 본인 배정 곡의 +합주 버튼 활성

- 예약 완료 키(`booking_completed_keys`) 일부만 완료 → 미완료 상태도 체험 가능

**시나리오 3 (최종 확정)**
- `is_final_schedule_confirmed=True`
- `PracticeSchedule` DB에 확정 저장 완료
- 일부 멤버의 ack 완료 처리 (멤버 확인 현황 보이도록)

## 4. 역할 배너 설계

### 4-1. 배너 구조

모든 데모 화면 최상단에 고정 표시한다. 역할에 따라 색상이 달라진다.

**매니저 뷰**
```
┌─────────────────────────────────────────────────────────────────────────────┐
│  🎛️  [데모 중] 매니저가 보는 화면입니다    [멤버 뷰로 전환]    [데모 종료]  │
└─────────────────────────────────────────────────────────────────────────────┘
  배경: 짙은 인디고(#3730A3) | 텍스트: 흰색
```

**일반 멤버 뷰**
```
┌─────────────────────────────────────────────────────────────────────────────┐
│  👤  [데모 중] 일반 멤버가 보는 화면입니다  [매니저 뷰로 전환]  [데모 종료]  │
└─────────────────────────────────────────────────────────────────────────────┘
  배경: 짙은 초록(#166534) | 텍스트: 흰색
```

### 4-2. 역할 전환 동작

- `[멤버 뷰로 전환]` / `[매니저 뷰로 전환]` 클릭 → `/demo/switch-role/` POST
- 세션의 `demo_role`을 `manager` ↔ `member`로 토글
- 현재 URL을 유지하며 새로고침 (같은 화면을 다른 역할로 재렌더)
- 역할 전환 시 배너 문구 + 색상 변경, 활성 버튼/편집 가능 영역 차이가 즉시 체감됨

### 4-3. 역할에 따른 UI 차이 (데모에서 명시적으로 보여줄 항목)

| 화면 요소 | 매니저 뷰 | 멤버 뷰 |
|---|---|---|
| 보드 편집 (드래그) | ✅ 가능 | ❌ 불가 (클릭 시 안내 토스트) |
| 리모콘 필터 | ✅ 전체 노출 | ❌ 사용 불가 버튼 숨김 |
| 공유/예약/확정 버튼 | ✅ 활성 | ❌ 숨김 |
| 불가능 오버레이 사유 | ✅ 상세 노출 | 제한적 노출 (프라이버시 정책) |
| 세션 배정 | ✅ 가능 | ❌ 지원만 가능 |
| 추가합주 (+합주) | ✅ 모든 곡 | ✅ 본인 배정 곡만 |

## 5. 튜토리얼(안내) 구현 방안

### 5-1. 목표 정의

튜토리얼은 사용자를 **강제로 끌고 다니지 않는다**. 대신:
- 처음 진입 시 주요 UI 요소를 하이라이트하며 1~2줄로 설명
- 사용자가 직접 클릭하거나 건너뛸 수 있다
- 설명한 기능이 어디에 있는지 **시각적으로 명확히 가리킨다**

### 5-2. 구현 방식: 단계별 스팟라이트 (Spotlight Tour)

별도 외부 라이브러리 없이 **순수 CSS/JS로 구현**한다. 이유:
- 기존 코드에 npm 의존성 추가 없이 경량 유지
- `match_result.html`처럼 복잡한 화면에서도 제어권 유지
- 라이브러리 버전 충돌 없음

#### 구현 원리

```
[스팟라이트 오버레이]
  - body에 반투명 검은 배경 div를 씌운다
  - 대상 요소의 위치를 getBoundingClientRect()로 계산
  - 해당 영역만 clip-path 또는 box-shadow로 구멍을 뚫어 밝게 표시
  - 설명 말풍선(tooltip)을 대상 요소 주변에 절대 위치로 배치

[네비게이션]
  - [다음] [건너뛰기] [전체 건너뛰기] 버튼
  - 진행 상태: "3 / 7"
```

#### CSS 스팟라이트 핵심 코드 아이디어

```css
.tutorial-overlay {
  position: fixed;
  inset: 0;
  z-index: 9000;
  pointer-events: none;
  /* box-shadow로 구멍 효과 */
  box-shadow: 0 0 0 9999px rgba(0, 0, 0, 0.65);
  border-radius: 4px;
  transition: all 0.3s ease;
}
.tutorial-tooltip {
  position: fixed;
  z-index: 9001;
  background: white;
  border-radius: 8px;
  padding: 16px;
  max-width: 320px;
  box-shadow: 0 4px 24px rgba(0,0,0,0.2);
}
```

#### JS 제어 흐름

```javascript
// tutorial_demo.js (신규 파일)
const TOUR_STEPS = [
  {
    target: '#quick-remote-panel',       // 리모콘
    title: '리모콘 (Quick Remote)',
    desc: '필터를 켜고 끄며 보드를 빠르게 탐색합니다. 매니저만 사용할 수 있습니다.',
    position: 'left',
  },
  {
    target: '.board-event:first-child',  // 합주 카드
    title: '합주 카드',
    desc: '드래그해서 시간대를 옮기거나, 엣지를 당겨 길이를 조절하세요.',
    position: 'right',
  },
  // ...
];

let currentStep = 0;

function showStep(index) {
  const step = TOUR_STEPS[index];
  const el = document.querySelector(step.target);
  if (!el) { currentStep++; showStep(currentStep); return; }
  positionOverlayOn(el);
  renderTooltip(step, el);
}

function positionOverlayOn(el) {
  const rect = el.getBoundingClientRect();
  const overlay = document.getElementById('tutorial-overlay');
  overlay.style.top = rect.top - 4 + 'px';
  overlay.style.left = rect.left - 4 + 'px';
  overlay.style.width = rect.width + 8 + 'px';
  overlay.style.height = rect.height + 8 + 'px';
}
```

### 5-3. 단계별 스팟라이트 시나리오 (시나리오 1 기준, 7스텝)

| Step | 대상 요소 | 설명 |
|---|---|---|
| 1 | 전체 보드 | "이것이 합주 스케줄 보드입니다. 주차별로 곡이 배치되어 있습니다." |
| 2 | 합주 카드 하나 | "카드를 드래그해서 시간대나 합주실을 옮길 수 있습니다." |
| 3 | 불가능 오버레이 | "빨간 패턴은 멤버가 불가능한 시간대입니다. 가능하면 이 시간은 피하세요." |
| 4 | 리모콘 패널 | "리모콘으로 강제배치/임시합주실/공유 후 변경 카드를 빠르게 필터할 수 있습니다." |
| 5 | 공유 버튼 | "조율이 끝나면 '공유' 버튼으로 멤버들에게 일정을 공개합니다." |
| 6 | 역할 배너 전환 버튼 | "'멤버 뷰로 전환' 버튼을 눌러 멤버가 보는 화면으로 바꿔보세요." |
| 7 | (완료) | "이제 직접 자유롭게 탐색해보세요! 모든 변경은 데모 종료 시 삭제됩니다." |

### 5-4. 튜토리얼 진입/스킵 정책

- 시나리오 첫 진입 시 자동 시작
- 우상단에 `[튜토리얼 건너뛰기]` 버튼 항상 표시
- `[전체 건너뛰기]` 클릭 → `localStorage.setItem('demo_tour_skipped', '1')` → 이후 재진입 시 자동 시작 안 함
- `[투어 다시 보기]` 버튼을 배너에 옵션으로 제공

## 7. 세션 데이터 구조

```python
request.session['demo_mode'] = True
request.session['demo_role'] = 'manager'          # 또는 'member'
request.session['demo_scenario'] = 1              # 1, 2, 3
request.session['demo_band_id'] = '<uuid>'
request.session['demo_meeting_id'] = '<uuid>'
request.session['demo_user_manager_id'] = '<pk>'  # 매니저 임시 계정
request.session['demo_user_member_id'] = '<pk>'   # 멤버 임시 계정
```

역할 전환은 `demo_role`만 바꾸고, `demo_user_manager_id`/`demo_user_member_id` 중 실제 `request.user`를 스왑한다. 뷰에서는 `request.user` 기준으로 기존 권한 검사가 동작하므로 별도 가드 불필요.

## 13. 주요 고려사항 및 주의점

### match_result.html 수정 시

- `match_result.html`은 9k+ 라인. 배너/오버레이 include는 파일 최상단에 `{% if is_demo %}` 블록 단 1회만 추가.
- 기존 핸들러 중복 바인딩 발생 금지. `tutorial_demo.js`는 별도 파일로 분리하여 조건부 로드.

### 권한 검사 일관성

- 역할 전환은 `request.user` 자체를 바꾸는 방식 사용.
- `is_demo` 컨텍스트를 보고 권한을 바이패스하는 코드를 절대 작성하지 않는다.
- 기존 `has_meeting_manager_permission()` 등 그대로 사용.

### 데이터 격리

- 데모 밴드 UUID를 세션에 보관. 기존 뷰의 `band` FK 체크가 자연스럽게 격리를 보장.
- 다른 실제 사용자 데이터에 접근하는 경로 없음.

### 모바일 대응

- 스팟라이트 투어는 모바일에서도 동작 (JS + CSS 기반).
- 드래그앤드롭 체험은 PC 권장. 모바일 감지 시 배너에 "드래그 기능은 PC에서 더 잘 보입니다" 1줄 추가.

### 튜토리얼과의 연계

- 데모 랜딩(`/demo/`)에 "단계별 튜토리얼 해보기" 링크를 명시.
- 튜토리얼 완료 화면에 "데모 시나리오 탐색하기" 역방향 링크를 추가.

### 16-3. 시나리오 검토 결과

#### 시나리오 A — 채택 (확장 필요)

> 60명 선곡회의, 80곡 중 25곡 선곡 후 합주 매칭

**적합한 이유:**
- 자동 매칭의 파워를 가장 직관적으로 보여줌
- 60명 규모가 "왜 이 서비스가 필요한가"를 설득함

**확인 필요 사항 (구현 전 선결):**
- "80곡에서 25곡 선곡" 과정이 앱 내 어떤 화면으로 체험되는지 명확하지 않음
- 현재 앱 플로우 기준으로 명시적인 "선곡" 기능은 없음. 세션 배정 완료 여부로 매칭 대상이 결정됨
- 데모에서 "80곡 등록, 25곡만 세션 배정 완료" 상태로 세팅하면 동작하나, 사용자가 선곡 과정을 직접 체험할 수 있는지 확인 필요

**구현 우려:**
- 60명 더미 데이터 생성 시 `_create_weekly_schedule_for_user` 60회 실행 → `demo_start` 응답 지연 가능성

**A에 포함해야 할 추가 항목:**
- 조율 단계에서 리모콘 필터 체험
- 강제 배치 카드 포함
- 공유 → 최종 확정 → 멤버 ack
- 최종 확정 이후 추가합주(`+합주`) 체험 (현재 어느 시나리오에도 없는 누락 기능)

---

#### 시나리오 B — 채택 (가장 독창적)

> 200명 대형 동아리, 6팀 각 4곡, 사용자는 1팀 매니저로 다른 팀과 예약 충돌 방지 체험

**적합한 이유:**
- 외부 미팅 충돌 감지(RoomBlock + 주황 오버레이)는 이 서비스 고유 기능
- "우리 팀만 잘 짜면 되는 게 아니라 다른 팀 예약도 피해야 한다"는 real-world 상황
- 매칭 보드에서 주황 오버레이가 많이 깔린 화면 자체가 시각적으로 임팩트 있음

**구현 방식:**

- 사용자는 6번째 팀 매니저로만 진입
- 임시합주실도 자연스럽게 포함 가능 (합주실 부족 상황 연출)

### 17-8. 다른 AI를 위한 작업 규칙

1. 시나리오 번호는 내부적으로 1/2/3을 유지한다. 표시(배너, 카드)만 A/B/C로 바꾼다.
